## Problem

`libs/draxul-nvim/src/mpack_codec.cpp:126â€“133` (`mpack_type_str`) and `135â€“143` (`mpack_type_bin`):

```cpp
uint32_t len = mpack_tag_str_length(&tag);
std::string text(len, '\0');   // allocates up to 4 GB before any validation
mpack_read_bytes(reader, text.data(), len);
```

`len` comes directly from the msgpack tag and is bounded only by `uint32_t` (~4 GB). The `std::string` constructor is called before mpack has a chance to validate that `len` bytes are actually available. A corrupt or malicious message with `len = 0xFFFFFFFF` causes a `std::bad_alloc` OOM crash. Although nvim is a trusted local process, fuzz harnesses and test pipes can inject arbitrary bytes.

**Trigger:** Feed corrupt msgpack with string tag `0xdb 0xff 0xff 0xff 0xff` (str32, 4 GB).

## Status

Added a 64 MB length guard (`kMaxMpackStringLen`) in `libs/draxul-nvim/src/mpack_codec.cpp` for both `mpack_type_str` and `mpack_type_bin`. When a tag claims more than 64 MB, the reader is flagged with `mpack_error_invalid` and `read_value` returns an empty/nil value before the `std::string` allocation, eliminating the `std::bad_alloc` OOM crash on `0xdb 0xff 0xff 0xff 0xff` and `0xc6 0xff 0xff 0xff 0xff`.

Added two non-slow guard tests in `tests/mpack_fuzz_tests.cpp` (`mpack guard: str32 claiming 4 GB does not OOM`, `mpack guard: bin32 claiming 4 GB does not OOM`). Both pass. `cmake --build build --target draxul draxul-tests` succeeds.
