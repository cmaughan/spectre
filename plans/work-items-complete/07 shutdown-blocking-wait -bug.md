# 07 shutdown-blocking-wait -bug

**Severity:** MEDIUM
**Type:** Bug
**Raised by:** Gemini
**Consensus:** review-bugs-consensus.md § M4

---

## Problem

Two shutdown paths block the main thread:

1. **`libs/draxul-nvim/src/nvim_process.cpp:378`** (`NvimProcess::shutdown()`, POSIX):
   ```cpp
   kill(impl_->child_pid_, SIGTERM);
   std::this_thread::sleep_for(std::chrono::microseconds(500000)); // 500 ms
   ```

2. **`libs/draxul-host/src/conpty_process.cpp:177`** (`ConPtyProcess::cleanup()`, Windows):
   ```cpp
   WaitForSingleObject(proc_info_.hProcess, 1000); // 1000 ms
   ```

Both are called on the main thread during tab close or application quit. CLAUDE.md requires non-blocking shutdown paths: "Keep shutdown paths non-blocking; a stuck Neovim child must not hang the UI on exit."

**Trigger:** Close a tab or quit the application while nvim is executing a long-running mapping, shell command, or blocking plugin call.

## Fix Strategy

Option A — background thread (simpler):
- [x] Move the timed wait + forceful kill into a short-lived `std::thread` that is detached. The process object becomes shared (or the thread captures a PID/HANDLE copy).
- [x] Main thread proceeds without blocking.

Option B — non-blocking state machine (cleaner):
- [ ] Add a `ShutdownState` enum to `NvimProcess` / `ConPtyProcess`: `Idle → SignalSent → ForcedKill → Done`.
- [ ] `shutdown()` sends SIGTERM/ClosePseudoConsole and records the deadline timestamp.
- [ ] A new `poll_shutdown()` method (called each frame from `App::pump_once()`) checks the deadline and escalates to SIGKILL / `TerminateProcess` if needed.
- [ ] `App::shutdown()` calls `shutdown()` then polls in a tight loop for at most 2 s before force-killing.

Option B is preferred per CLAUDE.md architecture guidance.

## Status

Fixed via Option A (detached background thread) on both POSIX and Windows shutdown paths.

- `libs/draxul-nvim/src/nvim_process.cpp` (`NvimProcess::shutdown`, POSIX): after sending `SIGTERM`, the 500 ms wait + `SIGKILL` escalation now runs in a detached `std::thread` capturing a copy of the child PID. The main thread returns immediately.
- `libs/draxul-host/src/conpty_process.cpp` (`ConPtyProcess::shutdown`, Windows): the `WaitForSingleObject(..., 1000)` + `TerminateProcess` + `CloseHandle` sequence is now run in a detached thread holding a copy of `hProcess`. The main thread no longer blocks waiting for the child to exit.

Build: `cmake --build build --target draxul` succeeds on macOS. The full `draxul-tests` target has a pre-existing compile error in `tests/rpc_codec_tests.cpp` (uses `MpackValue::as_uint`, owned by another work item) that is unrelated to this fix.
