## Problem

`libs/draxul-host/src/vt_parser.cpp:90–96` in `VtParser::feed()`, the buffer overflow path:

```cpp
if (plain_text_.size() >= kMaxPlainTextBuffer)
{
    DRAXUL_LOG_WARN(...);
    flush_plain_text();   // erases consumed bytes only
    plain_text_.clear();  // BUG: discards remaining partial UTF-8 bytes
}
plain_text_.push_back(ch);
```

`flush_plain_text()` (line 191) iterates clusters and calls `plain_text_.erase(0, offset)` to remove consumed bytes. Any incomplete multi-byte sequence at the tail of the buffer is intentionally left behind. The `plain_text_.clear()` on the next line then unconditionally discards those residual bytes, corrupting the next character.

This is a regression introduced when `02 vtparser-unbounded-buffers -bug.md` added the cap and the redundant `clear()` call.

**Trigger:** A stream of multi-byte UTF-8 characters (CJK, emoji) totalling exactly 64 KiB where a codepoint boundary falls at byte 65536.

## Status

Removed the redundant `plain_text_.clear()` call in `VtParser::feed()` after the
overflow `flush_plain_text()` call, so that any tail bytes belonging to an
incomplete UTF-8 sequence survive a cap-triggered flush and are completed by the
next feed instead of being silently discarded. The pre-existing cap check
(`plain_text_.size() >= kMaxPlainTextBuffer`) gating `push_back` still bounds
buffer growth, so the original unbounded-buffer issue does not regress.

Added a Catch2 regression test
`vtparser overflow: partial UTF-8 at cap boundary preserved` in
`tests/vtparser_overflow_tests.cpp` that streams >64 KiB of CJK text with the
final codepoint split across two `feed()` calls, asserting that every emitted
cluster equals the original 3-byte sequence (no mojibake from a discarded
partial). The test passes (21878 assertions); the full `[vtparser]` tag suite
also still passes (7 test cases, 21895 assertions). draxul builds cleanly.
