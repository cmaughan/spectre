# MpackValue::type() O(n) → O(1) Lookup Refactor

**Type:** refactor
**Priority:** 21
**Raised by:** Claude (#6 bad)
**Status:** open

## Problem

`MpackValue::type()` chains 9 `std::holds_alternative<T>` calls sequentially. The current code avoids using `storage.index()` directly to decouple the returned enum values from the variant's declaration order (with a comment explaining this). The approach is correct but produces 9 template instantiations on every type query in the hot RPC decode path.

A compile-time `std::array` mapping `storage.index()` → `MpackType` enum would be both O(1) and safe against reordering, as long as the mapping is checked with a `static_assert`.

- `libs/draxul-nvim/` — `MpackValue::type()` implementation (likely `mpack_value.h` or `mpack_codec.cpp`)
