## Problem

`libs/draxul-treesitter/src/treesitter.cpp:514–519, 717` in `CodebaseScanner::scan_thread()`:

```cpp
for (auto it = std::filesystem::begin(dir_iter), end = ...;
     it != end; it.increment(ec))
{
    if (ec || stop_flag_.load())
        break;          // early exit — scan is INCOMPLETE
    ...
}

snapshot->complete = true;   // BUG: always set, even after break
```

Whether the loop exited because it reached `end` (scan finished) or because of a filesystem error or stop-flag, `snapshot->complete` is always set to `true`. Consumers (e.g., the MegaCity host's dependency graph display) cannot distinguish a truncated result from a full one, leading to silently incorrect analysis.

**Trigger:** Any unreadable directory entry in the scanned tree: permissions error (`EACCES`), broken symlink, or NFS timeout.

## Fix Strategy

- [x] Track whether the loop completed without error:
  ```cpp

  snapshot->complete = !ec && !stop_flag_.load();
  ```

- [x] Log a WARN when scan is marked incomplete due to `ec`.

## Status

Fixed in `libs/draxul-treesitter/src/treesitter.cpp`. The scan loop's terminal
`snapshot->complete = true` now reads `snapshot->complete = !ec && !stopped`,
where `stopped` is captured from `stop_flag_.load()` after the loop. When the
loop exits because of a filesystem error, a `DRAXUL_LOG_WARN` (LogCategory::App)
is emitted with the `std::error_code` message so consumers can correlate the
truncated snapshot with the underlying cause. Cancellation via `stop_flag_` is
treated the same as an error for the `complete` flag — the optional `cancelled`
sub-flag suggested in the work item was not added since no consumer currently
distinguishes the two cases.

Build: `cmake --build build --target draxul draxul-tests` succeeded.
Tests: `./build/tests/draxul-tests "[treesitter]"` — all 5 cases / 52 assertions
pass. No new test was added: triggering a deterministic `std::error_code` from
`recursive_directory_iterator` on macOS is awkward (`skip_permission_denied` is
already set on the iterator, so chmod-000 is a no-op), and the existing
`wait_for_complete_snapshot` cases already cover the "normal scan still
complete=true" regression check.
