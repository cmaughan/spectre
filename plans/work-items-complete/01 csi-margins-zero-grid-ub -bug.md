## Problem

`libs/draxul-host/src/terminal_host_base_csi.cpp:475–476` in `csi_margins()`:

```cpp
const int top = std::clamp(param_or(params, 0, 1) - 1, 0, grid_rows() - 1);
const int bot = std::clamp(param_or(params, 1, grid_rows()) - 1, 0, grid_rows() - 1);
```

When `grid_rows() == 0`, the upper bound becomes `-1`. `std::clamp(x, 0, -1)` is undefined behavior — C++ requires `lo <= hi`. A zero-row grid can occur when the window is minimized or resized to zero height while the shell is redrawing its scroll region (`CSI r` / DECSTBM).

**Trigger:** Resize window to zero or near-zero height while a shell redraw is in flight.

- [x] Check `csi_scroll` and `csi_erase` for the same pattern — they may also call `grid_rows() - 1` without a guard.

---

## Fix Strategy

- [x] Add an early return guard at the top of `csi_margins()`:
  ```cpp
  if (grid_rows() <= 0 || grid_cols() <= 0) return;
  ```
- [x] Apply the same guard to any other CSI handler that uses `grid_rows() - 1` or `grid_cols() - 1` as a clamp bound.

## Status

Fixed `csi_margins()` in `libs/draxul-host/src/terminal_host_base_csi.cpp` by
adding an early return when `grid_rows() <= 0 || grid_cols() <= 0`. This is the
only CSI handler in the file that called `std::clamp(x, 0, grid_rows() - 1)`
without first wrapping the upper bound in `std::max(0, ...)`; every other
`grid_rows() - 1` / `grid_cols() - 1` use site in the file already guards with
`std::max(0, ...)` (lines 49, 79, 198, 203, 221, 232, 236, 237, 250, 266), so
no additional handlers needed the same fix.

Added a regression test in `tests/terminal_vt_tests.cpp`:
"terminal: DECSTBM on 1-row grid does not corrupt scroll region", which
exercises the smallest legal grid (rows=1; the public viewport API clamps to
>= 1) with both `\x1B[1;999r` and `\x1B[0;0r` sequences. Pre-existing DECSTBM
tests still pass. `cmake --build build --target draxul draxul-tests` succeeds
and `./build/tests/draxul-tests "[terminal]"` runs all 104 cases / 435
assertions cleanly.

Note on ASan acceptance criterion: the public host API (`on_viewport_changed`
and `apply_grid_size`) clamps rows/cols to >= 1, so reproducing a literal
zero-row state from a unit test would require backdoor access to private
members of `GridHostBase`. The defensive guard in the fix protects the
remaining race window — host receiving bytes before `apply_grid_size` runs
(initial `grid_rows_ = 0`) or `dependencies_available()` returning false
during teardown — without making the test fixture invasive.
