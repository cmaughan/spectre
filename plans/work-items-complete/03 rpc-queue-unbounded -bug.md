# Unbounded RPC Notification Queue

**Type:** bug
**Priority:** 03
**Raised by:** Claude, GPT, Gemini (unanimous)
**Status:** complete — bounded queue with drop-oldest semantics implemented; `belt-and-braces.md` updated

## Problem

`libs/draxul-nvim/src/rpc.cpp:24` accumulates incoming Neovim notifications in `std::vector<RpcNotification> notifications_` inside the `Impl` struct. The main thread drains it at line 175 via a swap-under-mutex pattern (`result.swap(impl_->notifications_)`). The reader thread pushes at line 286 (`impl_->notifications_.push_back(...)`) with no capacity check.

Under a burst of Neovim redraw events (large file open, `:%s/…/g` across thousands of lines, `:bufdo`, or a heavy plugin), the reader thread can outpace the main thread. Because the vector has no capacity limit, the queue grows without bound. The design document `belt-and-braces.md` names this as a known risk but no backpressure policy has been implemented.

Consequences:
- Unbounded memory growth during bursts.
- Increasing drain latency as the vector grows.
- Potential frame drops as the main thread spends an entire frame processing a large batch.

## Affected Files

- `libs/draxul-nvim/src/rpc.cpp` — notification accumulation and drain logic
- `libs/draxul-nvim/include/draxul/nvim_rpc.h` (or equivalent) — public queue interface

## Implementation Plan

- [x] Define a bounded capacity constant, e.g. `kMaxNotificationQueueDepth = 4096`.
- [x] In the reader thread push path: if `pending_.size() >= kMaxNotificationQueueDepth`, drop the oldest notification (ring-buffer semantics).
- [ ] **Optional follow-up:** coalescing. For redraw notifications, if multiple `flush` events are queued, keep only the last one (earlier flush events are superseded by later grid state). This is compatible with the existing grid dirty-tracking because the grid already accumulates all intermediate mutations.
- [x] Add `DRAXUL_LOG_WARN` when the queue depth exceeds a threshold (e.g., 512) so the condition is visible in logs.
- [ ] Expose queue depth as a metric in the ImGui diagnostics panel (accessor `notification_queue_depth()` is now available).
- [x] Update `belt-and-braces.md` to mark this risk as addressed.
