## Problem

`libs/draxul-host/src/terminal_host_base.cpp:56–72`, `TerminalHostBase::pump()`:

```cpp
do {
    for (const auto& chunk : chunks)
        consume_output(chunk);
    chunks = do_process_drain();
} while (!chunks.empty());
```

The loop continues draining until the child produces no more output. If a process generates output faster than it can be parsed (e.g., `yes`, `cat /dev/urandom`, or `cat` of a large file), the main thread is trapped here indefinitely. The SDL event queue is never pumped, making the application unresponsive.

**Trigger:** Run `yes | head -c 500M` or `cat /dev/urandom` in any shell pane.

## Fix Strategy

- [x] Capture wall time before the loop:
  ```cpp
  const auto deadline = std::chrono::steady_clock::now() + std::chrono::milliseconds(8);
  ```
- [x] Change the `while` condition to also check the deadline:
  ```cpp
  } while (!chunks.empty() && std::chrono::steady_clock::now() < deadline);
  ```
- [x] Log a DEBUG message when the budget is exceeded so throughput issues are observable.
- [x] Verify that `flush_grid()` is still called even when the budget expires (it is — it's called after the loop regardless).

## Status

Fixed in `libs/draxul-host/src/terminal_host_base.cpp` (`TerminalHostBase::pump`).
Added an 8 ms wall-clock deadline (`std::chrono::steady_clock`) around the
re-drain loop. After processing each batch the loop checks
`steady_clock::now() >= deadline` and breaks early; any remaining producer
output is picked up on the next frame, so runaway producers like
`yes`/`cat /dev/urandom` can no longer trap the main thread and starve the
SDL event loop. When the budget trips, a `DRAXUL_LOG_DEBUG` line under
`LogCategory::App` is emitted so throughput issues remain observable.
`flush_grid()` is still called after the loop, regardless of whether the
budget expired. `draxul-host` and the `draxul` app target build cleanly with
the change.
