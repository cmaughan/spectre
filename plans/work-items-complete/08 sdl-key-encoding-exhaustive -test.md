## Problem / Gap

`encode_key_tests.cpp` covers common SDL keycodes but not the full SDL3 keycode space. When SDL3 updates its keycode enum, regressions in `NvimInput::encode_key` can go undetected until a user reports a broken keybinding.

## Affected Files

- `tests/encode_key_tests.cpp` — extended with exhaustive data-driven tests
- `libs/draxul-host/src/terminal_key_encoder.cpp` — encoder under test (read-only)

## Test Plan

- [x] Data table covering all 26 special/navigation/F-key mappings — each asserted to produce the exact expected VT sequence.
- [x] Ctrl+a through Ctrl+z loop — asserts control characters 0x01-0x1A.
- [x] Alt+printable-ASCII loop (0x20-0x7E) — asserts ESC-prefixed character for each.
- [x] Printable ASCII without modifier — asserts empty (intentionally handled by SDL text-input path).
- [x] Modifier-only keys (LSHIFT, RSHIFT, LCTRL, RCTRL, LALT, RALT, LGUI, RGUI, CAPSLOCK) — asserts empty.
- [x] F13-F24 — asserts empty with documentation that these are intentionally unmapped.
- [x] F1-F12 with Shift/Ctrl/Alt modifier combos — pins current behaviour (unmodified base sequence) as a regression baseline.
- [x] Numpad keys (KP_0–KP_9, KP_ENTER, KP_PLUS, KP_MINUS, KP_MULTIPLY, KP_DIVIDE, KP_PERIOD) — asserts empty, documents gap.

## Implementation Notes

- Tests call `encode_terminal_key()` directly (from `draxul-host`) rather than going through the full `KeyTestSetup` host, so they are fast and independent of SDL window state.
- `draxul::sdl::translate_key()` (from item 18 `sdl_event_translator.h`) is the SDL boundary; `encode_terminal_key()` is the pure VT logic layer being tested.
- The "F13-F24 unmapped" and "numpad unmapped" tests intentionally document gaps so future additions are a test update, not a silent hole.
