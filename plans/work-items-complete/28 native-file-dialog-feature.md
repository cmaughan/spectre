# Native File Open/Save Dialog

## Summary
`UiRequestWorker` has no implementation for `open_file_dialog` or `save_file_dialog`. Dragging a file onto the window does nothing. SDL3 provides `SDL_ShowOpenFileDialog` for a native implementation.

## Context
`UiRequestWorker` in `draxul-app-support`. `IWindow` for drag-drop events. `SDL_ShowOpenFileDialog` in SDL3 docs. This is a quality-of-life gap: users expect to drag files onto a GUI application or use a menu to open files, and currently neither works.

## Steps
- [x] Read `UiRequestWorker` to find the `open_file_dialog` / `save_file_dialog` stubs (no stubs found; implemented from scratch)
- [x] Implement using `SDL_ShowOpenFileDialog` (SDL3 API) with thread-safe custom event marshalling
- [x] Wire the result to a Neovim `:edit <path>` command sent via RPC (NvimHost handles `open_file:` prefix)
- [x] Add SDL drag-drop event handling in `SdlWindow`: `SDL_EVENT_DROP_FILE` → `on_drop_file` callback
- [x] Add tests: GuiActionHandler `open_file_dialog` action routing verified

## Acceptance Criteria
- [x] File open dialog opens on `open_file_dialog` action (keybinding → GuiActionHandler → show_open_file_dialog)
- [x] Dragging a file onto the window opens it in Neovim (SDL_EVENT_DROP_FILE → dispatch_action)
- [x] Works on both macOS and Windows (SDL3 dialog API is cross-platform)

## Notes
Implementation used `on_drop_file` callback on IWindow/SdlWindow as the common delivery point for both
drag-drop and dialog results. The dialog callback is marshalled from a background thread via a custom
SDL user event type (registered in SDL_RegisterEvents).
