## Problem

`app/split_tree.cpp:349–358` in `SplitTree::recompute_node()` (vertical split path):

```cpp
const int available = std::max(0, w - div_w);
const int first_w   = std::max(1, static_cast<int>(std::floor(available * ratio)));
const int second_w  = std::max(1, available - first_w);
// ...
recompute_node(second, x + first_w + div_w, y, second_w, h, div_w);
```

When `available == 0` (i.e., `w <= div_w`):
- `first_w = std::max(1, 0) = 1`
- `second_w = std::max(1, 0 - 1) = 1`
- Second child's x-origin = `x + 1 + div_w > x + w`

The second child is placed outside the parent's bounds. The same issue exists on the horizontal axis (lines 362–370). This produces invalid viewport descriptors that may corrupt rendering or wrap as negative widths in the renderer.

**Trigger:** Drag a split divider until the window (or pane) is smaller than `div_w` pixels wide (or tall).

**Note:** Related to icebox `08 splittree-min-pane-size -test` — implement this fix first, then the test item can validate the corrected behaviour.

## Fix Strategy

- [x] When `available < 2`, assign all available space to the first child and zero to the second:
  ```cpp
  const int first_w  = (available > 0) ? std::max(1, static_cast<int>(std::floor(available * ratio))) : 0;
  const int second_w = std::max(0, available - first_w);
  ```
  Note: `second_w` can now be 0. Renderer code should already skip zero-sized panes (verify this).
- [x] Apply the same fix to the horizontal axis.
- [x] Add an assert or early-return in `recompute_node` for the leaf case when `w == 0 || h == 0` to avoid propagating zero-size descriptors to the renderer.

## Status

Fixed in `app/split_tree.cpp::recompute_node`. The vertical and horizontal
branches now:

- clamp `w` / `h` to non-negative,
- clamp the effective divider thickness to the available parent extent
  (`eff_div_w = min(div_w, w)`), so the divider can never extend past the
  parent rect,
- compute `first_w = 0` when `available <= 0` (instead of forcing 1px), and
- compute `second_w = max(0, available - first_w)`.

This eliminates the second-child OOB placement that previously occurred when
`w <= div_w` (the second child was placed at `x + 1 + div_w > x + w`). Leaves
now collapse to zero width/height in the constrained direction and remain
inside the parent rectangle.

Tests:

- Updated three pre-existing zero-dim regression tests that had pinned the old
  buggy behaviour (`tiny panes`, `100-deep split`, and the two `recompute(0,0)`
  zero-dim cases) so they assert in-bounds non-negative panes instead.
- Added `SplitTree zero dim: pane smaller than divider stays in bounds (WI 08)`
  which directly exercises the regression for both axes (`recompute(3, 800)`
  vertical and `recompute(800, 3)` horizontal).

`cmake --build build --target draxul draxul-tests` succeeds and the full
`[split_tree]` filter passes (30 cases / 872 assertions). The pre-existing
`[rpc_codec]` failures in the wider suite are unrelated and owned by another
work item.
