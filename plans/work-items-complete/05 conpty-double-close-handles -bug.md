## Problem

`libs/draxul-host/src/conpty_process.cpp:133–134` in `ConPtyProcess::spawn()`:

```cpp
CloseHandle(pty_input_read);    // required before CreateProcess
CloseHandle(pty_output_write);  // required before CreateProcess
// ... local vars NOT set to INVALID_HANDLE_VALUE ...

if (!created)
{
    cleanup();   // lambda re-closes pty_input_read and pty_output_write!
    return false;
}
```

The local `cleanup` lambda (lines 66–80) guards with `!= INVALID_HANDLE_VALUE`, but since the local variables are not nulled after the mandatory pre-CreateProcess closes, the lambda closes them again if `CreateProcessA` fails. Double-closing a `HANDLE` corrupts the Windows kernel handle table and can cause subsequent `CloseHandle` calls in unrelated code to close the wrong handle.

**Trigger:** `CreateProcessA` fails during spawn (e.g., command not found, invalid working directory).

## Fix Strategy

- [x] Null the local variables immediately after the unconditional closes:
  ```cpp
  CloseHandle(pty_input_read);  pty_input_read  = INVALID_HANDLE_VALUE;
  CloseHandle(pty_output_write); pty_output_write = INVALID_HANDLE_VALUE;
  ```
- [x] No other changes needed — the lambda's `!= INVALID_HANDLE_VALUE` guard then correctly skips these handles.
