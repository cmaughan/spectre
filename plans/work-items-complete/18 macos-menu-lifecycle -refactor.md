## Problem

`app/macos_menu.mm` stores menu action callbacks in global mutable state (likely a static `std::function` map or global Objective-C objects). This:

- Makes lifecycle ownership implicit — the callbacks capture raw `GuiActionHandler*` references that may be destroyed before the menu is torn down.
- Blocks multi-window support (a second window would need its own menu context, but global state is shared).
- Makes the macOS menu hard to test — there is no way to inject a fake `GuiActionHandler` or verify which menu items exist without actually running an NSApplication event loop.

---

## Steps

- [ ] Read `app/macos_menu.mm` in full (and any associated header).
- [ ] Identify all global/static state: static function maps, static Objective-C delegates, static `GuiActionHandler*` captures.
- [ ] Refactor toward an instance-owned `MacOsMenu` class:
  1. Replace global state with instance members.
  2. Constructor takes `GuiActionHandler&` by reference (or as a `shared_ptr`).
  3. Destructor clears all NSMenu items pointing back at the instance.
  4. `App` owns a `std::unique_ptr<MacOsMenu>` with a lifetime tied to the app.
- [ ] If the action table is hardcoded, make it data-driven from `GuiActionHandler::registered_actions()` (or equivalent) so new actions automatically appear in the menu without manual wiring.
- [ ] Run `cmake --build build --target draxul`.
- [ ] Smoke-test: launch the app, open the macOS menu bar, verify all actions still appear and work.
- [ ] Run `clang-format` on touched files.
