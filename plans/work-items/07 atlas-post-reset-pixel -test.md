# Atlas Post-Reset Pixel Correctness Test

**Type:** test
**Priority:** 07
**Raised by:** Claude
**Status:** open

## Problem / Gap

`ligature_atlas_reset_tests.cpp` verifies that the atlas reset flow runs without assertion failures. However it does not verify pixel-level correctness: that after a reset-and-re-upload cycle, the glyphs rendered to the screen are visually identical to those rendered before the overflow.

A glyph-region remapping bug after atlas reset could produce incorrect pixel output that only surfaces in a render snapshot comparison.

## Affected Files

- `libs/draxul-font/` — `GlyphCache`, atlas reset logic
- `tests/ligature_atlas_reset_tests.cpp` — existing test to extend or companion
- `tests/render/` — render snapshot reference images

## Test Plan

- [ ] Identify or create a render scenario that exercises the atlas enough to trigger an overflow: many distinct glyphs (Unicode + ligatures) that exceed the 2048×2048 atlas capacity when all loaded simultaneously.
- [ ] Capture a render snapshot before the atlas reset.
- [ ] Force an atlas reset (either via the existing test hook or by filling it with throw-away glyphs).
- [ ] Render the same glyph set again and capture a second snapshot.
- [ ] Compare pixel-by-pixel (or using the existing `bless` comparison infrastructure). Assert images match within a tolerance threshold.
- [ ] Add this as a render scenario `.toml` file in `tests/render/` so it can be run via `py do.py` and blessed.

## Implementation Notes

- The render test infrastructure already supports toml-based scenario capture and comparison. Use that pattern.
- If no test hook exists to force an atlas reset without filling it naturally, add a `reset_for_test()` method behind `#ifdef DRAXUL_TEST` or as an injectable in the test seam.

## Dependencies

- Should be run as part of the existing `ctest` / `py do.py` render-test suite.
