# Open File Path Injection Bug

**Type:** bug
**Priority:** 00 (highest)
**Raised by:** GPT (unique finding)
**Status:** open

## Problem

`input_dispatcher.cpp:111` forwards raw file paths from drop events and open-file dialogs as `"open_file:" + path`. `libs/draxul-host/src/nvim_host.cpp:156` converts that string into a Neovim command `nvim_command("edit " + path)`.

Paths containing spaces, quotes (`"`), pipe characters (`|`), percent signs (`%`), or command-like substrings can be misparsed by Neovim's command parser or do the wrong thing entirely. A dropped path like `/home/user/my file.txt` becomes `:edit /home/user/my file.txt` which Neovim interprets as editing `/home/user/my` with `file.txt` as a trailing argument error. A path with `|` can inject a second Vimscript command.

## Affected Files

- `app/input_dispatcher.cpp` — line ~111, path forwarding
- `libs/draxul-host/src/nvim_host.cpp` — line ~156, edit command construction

## Implementation Plan

- [ ] In `nvim_host.cpp`, replace `nvim_command("edit " + path)` with a Neovim API call via `nvim_buf_call` or `nvim_exec_lua` that passes the path as a Lua string argument (not interpolated into a command string). Example: `nvim_exec_lua("vim.cmd.edit(vim.fn.fnameescape(...))", {path})`
- [ ] Alternatively use `nvim_command("edit " + fnameescape(path))` where `fnameescape` is called via the Neovim API before building the command string.
- [ ] Prefer the Lua/API approach as it avoids the escaping problem entirely.
- [ ] Review `input_dispatcher.cpp` for any other sites that construct RPC command strings from user-supplied paths.
- [ ] Add a `fnameescape` or equivalent helper in `nvim_host.cpp` if the API call approach is not feasible.
