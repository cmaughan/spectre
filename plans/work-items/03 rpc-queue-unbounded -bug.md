# Unbounded RPC Notification Queue

**Type:** bug
**Priority:** 03
**Raised by:** Claude, GPT, Gemini (unanimous)
**Status:** open — documented known risk in `plans/design/belt-and-braces.md`

## Problem

`libs/draxul-nvim/src/rpc.cpp:23` accumulates incoming Neovim notifications in a `std::vector<RpcNotification>`. The main thread drains it opportunistically at line ~175 via a swap-under-mutex pattern.

Under a burst of Neovim redraw events (large file open, `:%s/…/g` across thousands of lines, `:bufdo`, or a heavy plugin), the reader thread can outpace the main thread. Because the vector has no capacity limit, the queue grows without bound. The design document `belt-and-braces.md` names this as a known risk but no backpressure policy has been implemented.

Consequences:
- Unbounded memory growth during bursts.
- Increasing drain latency as the vector grows.
- Potential frame drops as the main thread spends an entire frame processing a large batch.

## Affected Files

- `libs/draxul-nvim/src/rpc.cpp` — notification accumulation and drain logic
- `libs/draxul-nvim/include/draxul/nvim_rpc.h` (or equivalent) — public queue interface

## Implementation Plan

- [ ] Define a bounded capacity constant, e.g. `kMaxNotificationQueueDepth = 4096`.
- [ ] In the reader thread push path: if `pending_.size() >= kMaxNotificationQueueDepth`, either drop the oldest notification (ring-buffer semantics) or block the reader thread briefly with a condition variable until the main thread has drained below a low-water mark.
- [ ] **Recommended policy:** coalescing. For redraw notifications, if multiple `flush` events are queued, keep only the last one (earlier flush events are superseded by later grid state). This is compatible with the existing grid dirty-tracking because the grid already accumulates all intermediate mutations.
- [ ] Add `DRAXUL_LOG_WARN` when the queue depth exceeds a threshold (e.g., 512) so the condition is visible in logs.
- [ ] Expose queue depth as a metric in the ImGui diagnostics panel.
- [ ] Update `belt-and-braces.md` to mark this risk as addressed.

## Test Plan

- [ ] Unit test: push `kMaxNotificationQueueDepth + 1` notifications, assert the reader does not block indefinitely and the drain produces at most `kMaxNotificationQueueDepth` items (or coalesced equivalent).
- [ ] This enables un-iceboxing work item `18 rpc-burst-transport` once the policy is defined.
