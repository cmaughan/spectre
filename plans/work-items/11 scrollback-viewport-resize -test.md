## Problem

When the terminal is resized to fewer columns, scrollback lines that are wider than the new width must be handled correctly — either truncated, wrapped, or preserved as-is depending on the implementation. Currently there is no test that resizes the grid after filling scrollback, leaving index corruption and out-of-bounds access as latent risks.

---

## Steps

- [ ] Read `libs/draxul-host/src/scrollback_buffer.cpp` (or equivalent). Understand how line storage and viewport indices work.
- [ ] Read existing scrollback tests in `tests/`.
- [ ] Write a test using `replay_fixture.h` or a direct scrollback harness:
  1. Fill scrollback with 50 lines of 80 columns of text.
  2. Resize the grid to 40 columns.
  3. Assert: no out-of-bounds access (test should not ASAN-fault or throw).
  4. Assert: scrollback viewport reads the expected line count — not corrupted or doubled.
  5. Assert: scrolling to the top of scrollback still returns valid content.
- [ ] Add a resize-back-up test: resize from 40 → 80 columns; verify lines are still accessible.
- [ ] Run `cmake --build build --target draxul-tests && ctest`.
- [ ] Run `clang-format` on touched files.

## Acceptance Criteria

- [ ] No out-of-bounds access after resize (clean under ASan).
- [ ] Scrollback viewport reports correct line count after resize.
- [ ] Round-trip resize (narrow then wide) leaves scrollback navigable.
