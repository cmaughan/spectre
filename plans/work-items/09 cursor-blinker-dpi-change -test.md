# Cursor Blinker DPI Change Test

**Type:** test
**Priority:** 09
**Raised by:** Claude
**Status:** open

## Problem / Gap

`cursor_blinker_tests.cpp` tests cursor blink timing at a fixed DPI but does not test what happens when the display DPI changes mid-session (e.g., moving the window to a monitor with a different pixel density). Blink rate or blink period could be incorrectly calculated after a DPI change, causing the cursor to freeze or blink at the wrong rate.

## Affected Files

- `tests/cursor_blinker_tests.cpp` — existing test to extend
- `libs/draxul-app-support/` — `CursorBlinker` implementation

## Test Plan

- [ ] In `cursor_blinker_tests.cpp`, add a test that:
  1. Constructs a `CursorBlinker` with an initial DPI (e.g. 96 PPI).
  2. Advances simulated time to verify the cursor toggles at the expected period.
  3. Changes the DPI (e.g. to 192 PPI, simulating a Retina display).
  4. Asserts that the cursor blink period is recalculated correctly (not frozen at the old period, not jumping to a wrong period).
  5. Advances time further and verifies the new period is stable.
- [ ] Test: DPI decrease (from high-DPI to standard) also recalculates correctly.
- [ ] Test: DPI change of exactly 1.0 (no-op) does not reset the blink timer unnecessarily.

## Implementation Notes

- Use the existing `FakeClock` or time-injection pattern from `cursor_blinker_tests.cpp` to keep the test deterministic.
- If `CursorBlinker` does not currently expose a `set_dpi` or `on_dpi_changed` method, add one as part of this work item and wire it into the call site in `App`.
