## Problem

`rpc_integration_tests.cpp` tests the RPC decoder, but may not exercise msgpack messages that span multiple pipe reads at OS pipe-buffer boundaries (4 KB, 16 KB, 64 KB). A large `grid_line` batch (e.g., 70 KB of cell updates) split across several reads must be reassembled correctly by the stream accumulator in `NvimRpc`. A regression here causes silent partial-frame rendering with no error.

- [ ] Read `libs/draxul-nvim/src/nvim_rpc.cpp` (or equivalent) to understand how incoming bytes are accumulated and dispatched to the MPack decoder.
- [ ] Read `tests/rpc_integration_tests.cpp` to see what fragmentation cases already exist.
- [ ] Add a test that:
  1. Constructs a large `grid_line` msgpack notification well above 64 KB (simulate ~200 cells × ~350 bytes each, or build a realistic oversized batch).
  2. Feeds the bytes to the accumulator in chunks of 1 KB (simulating OS-level fragmentation).
  3. Asserts that the decoder emits exactly one complete notification with the expected cell count — not partial frames, not duplicate frames.
