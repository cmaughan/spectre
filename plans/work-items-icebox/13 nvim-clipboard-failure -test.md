## Problem

The Neovim clipboard provider is injected via embedded Lua strings inside `NvimHost`. Failure paths — `nvim_get_api_info` returning an unexpected shape, `clipboard_set` RPC timeout, or a shutdown request racing with an in-flight clipboard RPC — are untested. These are the exact scenarios that cause silent clipboard loss in real terminal sessions.

---

## Steps

- [ ] Read `libs/draxul-nvim/src/nvim_host.cpp` (or equivalent). Find clipboard provider injection and `clipboard_set`/`clipboard_get` RPC calls.
- [ ] Read existing clipboard-related tests if any exist in `tests/`.
- [ ] Write test cases using a `FakeNvimRpc` stub (or extend the existing fake if one exists):
  1. `nvim_get_api_info` returns an empty or malformed response → clipboard provider injection fails cleanly with a WARN log, not a crash.
  2. `clipboard_set` RPC times out (stub returns no response) → caller gets a timeout/error, not a hang.
  3. Shutdown is requested while `clipboard_set` is in flight → no use-after-free, no deadlock; the in-flight request is cancelled cleanly.
  4. `clipboard_get` returns an unexpected msgpack type (string instead of array) → graceful error, clipboard returns empty string.
- [ ] Run `cmake --build build --target draxul-tests && ctest`.
- [ ] Run `clang-format` on touched files.
